Security threat analysis — deep dive
Why this section matters more than design. Richgyan India manages data for 50,000+ students, 11,000+ trainers, and franchise centers across India. A single breach can expose every student's PII, allow certificate forgery at scale, destroy employer trust built over 8 years, and invite regulatory action under India's DPDP Act 2023. Every threat below has been identified from the live richgyan.com stack.
Certificate Forgery & Verification Bypass
Anyone can fake a Richgyan certificate — there's no database to check against
Critical
🔍 How this attack works
Your current certificate verification page at /certificate-verification/ is a static HTML page — it has no backend database query. An attacker can open any image editor, duplicate a real certificate template, change the name, dates, and course name, and print a fully convincing fake. There is zero technical barrier. Additionally, since there's no cryptographic signature or tamper-evident hash, even a real certificate can be altered after issuance.
💥 Impact on Richgyan India
  • Fake certificate holders get jobs using Richgyan's name — destroying your brand with employers
  • Legitimate students lose credibility when employers can't verify their certificates
  • Franchise centers can issue unauthorized certificates with no oversight
  • Legal liability if forged certificates are used for government job applications
✅ Our Solution
  • Each certificate gets a unique UUID stored in a PostgreSQL database at issuance time
  • A SHA-256 cryptographic hash of all certificate fields is stored — any field change invalidates it
  • QR code printed on every certificate links to richgyan.com/verify/[UUID] — instant verification
  • Employer-facing public API: GET /api/v1/verify?cert_id=xxx returns verified JSON
  • PDF certificates generated with digital signature using ReportLab + Python cryptography library
PostgreSQL UUID SHA-256 Hash QR Code Generation ReportLab PDF Sign Public Verify API
Broken Authentication & Session Hijacking
No OTP, no 2FA — student & admin accounts fall to a single phishing email
Critical
🔍 How this attack works
The current system uses WordPress native login — username + password only. Attackers exploit this via: (1) Credential stuffing — using leaked password databases, (2) Phishing — sending a fake "Richgyan login" page, (3) Session fixation — WordPress session cookies don't rotate on login. Your exposed /my-account/ and /richgyan endpoints have no rate limiting, meaning an attacker can try thousands of passwords per minute.
💥 Impact on Richgyan India
  • Admin account takeover exposes all 50,000+ student records instantly
  • Franchise center accounts compromised — fake enrollment, fee fraud
  • No audit log means you wouldn't even know a breach happened
✅ Our Solution
  • OTP-first login via Twilio SMS — phone number + OTP, no password required for students
  • JWT with refresh token rotation — every session refresh invalidates the old token
  • Redis rate limiter — max 5 failed OTP attempts per phone per 15 minutes, then 1-hour lockout
  • Optional TOTP 2FA (Google Authenticator) for Admin and Franchise owners
Twilio OTP JWT Rotation Redis Rate Limit RBAC TOTP (pyotp)
SQL Injection & WordPress Database Exposure
Unpatched WordPress plugins provide direct access to your entire student database
Critical
🔍 How this attack works
WordPress plugins — especially WooCommerce, form plugins, and LMS plugins — have a long history of SQL injection vulnerabilities. Automated scanners (like SQLMap) probe /?s=, /product/?add-to-cart=, and form endpoints continuously. A single vulnerable plugin parameter allows an attacker to dump the entire wp_users and wp_usermeta tables — including hashed passwords, emails, phone numbers, and payment history for all students and franchise owners.
✅ Our Solution
  • All database queries use parameterized statements via SQLAlchemy ORM — injection impossible by design
  • Cloudflare WAF rule set blocks common SQLi patterns at the edge before they reach your server
  • Database user has principle of least privilege — API can't DROP or ALTER tables
SQLAlchemy ORM Parameterized Queries Cloudflare WAF Least Privilege DB User
Malicious File Upload & RCE (Remote Code Execution)
WordPress media upload endpoint accepts PHP files — a direct server takeover vector
High
🔍 How this attack works
WordPress's /wp-admin/media-new.php and various plugin upload endpoints are common targets for PHP webshell uploads. If a misconfigured plugin allows .php file uploads, the attacker executes arbitrary code on your server with the web server's privileges — typically leading to full server compromise, data exfiltration, and ransomware deployment.
✅ Our Solution
  • All file uploads validated by MIME type + magic bytes — extension spoofing blocked
  • Uploads stored in Cloudinary / AWS S3 — never on the application server, never executable
  • File size limits enforced server-side on every endpoint
MIME + Magic Byte Check S3/Cloudinary Storage Server-side File Limits
Student PII Data Breach (DPDP Act 2023 Risk)
50,000+ student records with no encryption at rest — a breach means ₹250 crore penalties
Critical
🔍 How this attack works
Student records in the current WordPress/WooCommerce database include full names, phone numbers, addresses, email IDs, course names, fee payment history — stored in plain text or with only bcrypt on passwords. India's Digital Personal Data Protection (DPDP) Act 2023 imposes penalties up to ₹250 crore per violation for significant data fiduciaries managing large-scale student data.
✅ Our Solution
  • AES-256 encryption at rest for all sensitive PII fields using pgcrypto extension
  • DPDP-compliant consent system — explicit consent checkboxes at enrollment, stored with timestamp and IP
  • Automated daily encrypted backups to AWS S3 with server-side encryption
AES-256 at Rest pgcrypto DPDP Consent System S3 Encrypted Backup
DDoS Attacks & No CDN / Rate Limiting
A competitor or malicious actor can flood your server and take you offline for days
Medium
🔍 How this attack works
Your current setup has no CDN — meaning all traffic hits your origin server directly. With no rate limiting and no WAF, even a moderate botnet (5,000 requests/second) would bring WordPress down. DDoS attacks in Indian ed-tech frequently target enrollment periods and result announcement days.
✅ Our Solution
  • Cloudflare Free/Pro tier in front of all traffic — absorbs DDoS attacks up to terabit scale
  • Cloudflare CDN caches static assets at 300+ edge locations globally
  • Redis rate limiting on all API endpoints — 100 req/min per IP
Cloudflare WAF + DDoS Cloudflare CDN Redis Rate Limit
